# Feature Specification: Board Management

**Feature Branch**: `003-board-management`  
**Created**: 2026-03-02  
**Status**: Draft  
**Input**: User description: "Необходимо реализовать возможность создавать доски. Доска должна состоять из нескольких колонок, названия которых соответствуют статусам задач - к выполнению, в работу, возникла проблема, сделано, отмена"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Создание доски с предопределёнными колонками (Priority: P1)

Пользователь может создать новую доску для организации своих задач. При создании доски автоматически создаются пять колонок с предопределёнными названиями в порядке: "К выполнению", "В работу", "Возникла проблема", "Сделано", "Отмена". Пользователь указывает название доски при создании.

**Why this priority**: Это базовая функциональность, без которой невозможно использовать систему управления задачами. Создание доски - это первый шаг для организации работы.

**Independent Test**: Можно полностью протестировать создание доски с предопределёнными колонками, создав одну доску и проверив, что все пять колонок созданы с правильными названиями.

**Acceptance Scenarios**:

1. **Given** пользователь авторизован, **When** пользователь создаёт доску с названием "Мой проект", **Then** создаётся доска с названием "Мой проект" и пятью колонками в порядке: "К выполнению", "В работу", "Возникла проблема", "Сделано", "Отмена"
2. **Given** пользователь авторизован, **When** пользователь создаёт доску с пустым названием, **Then** система возвращает ошибку валидации с сообщением о том, что название обязательно
3. **Given** пользователь авторизован, **When** пользователь создаёт доску с названием, которое уже существует у этого пользователя, **Then** система возвращает ошибку с сообщением о том, что доска с таким названием уже существует

---

### User Story 2 - Просмотр списка своих досок (Priority: P2)

Пользователь может просмотреть список всех созданных им досок. Каждая доска отображается с её названием и количеством задач в каждой колонке.

**Why this priority**: Пользователи должны иметь возможность навигации между своими досками. Это важная функциональность для управления несколькими проектами.

**Independent Test**: Можно протестировать просмотр списка досок, создав несколько досок и проверив, что все они отображаются в списке с правильной информацией.

**Acceptance Scenarios**:

1. **Given** пользователь авторизован и имеет 3 доски, **When** пользователь запрашивает список своих досок, **Then** отображаются все 3 доски с их названиями и количеством задач в каждой колонке
2. **Given** пользователь авторизован и не имеет досок, **When** пользователь запрашивает список своих досок, **Then** отображается пустой список с сообщением о том, что досок нет
3. **Given** пользователь авторизован, **When** пользователь запрашивает список досок другого пользователя, **Then** возвращается только список собственных досок пользователя

---

### User Story 3 - Удаление доски (Priority: P3)

Пользователь может удалить созданную им доску. При удалении доски удаляются все связанные с ней колонки и задачи.

**Why this priority**: Удаление досок необходимо для управления жизненным циклом проектов и очистки от ненужных данных. Это менее критичная операция, чем создание и просмотр.

**Independent Test**: Можно протестировать удаление доски, создав доску с задачами и проверив, что после удаления доска и все связанные данные недоступны.

**Acceptance Scenarios**:

1. **Given** пользователь авторизован и имеет доску "Тестовая", **When** пользователь удаляет эту доску, **Then** доска "Тестовая" и все связанные с ней колонки и задачи удаляются
2. **Given** пользователь авторизован, **When** пользователь пытается удалить несуществующую доску, **Then** возвращается ошибка с сообщением о том, что доска не найдена
3. **Given** пользователь авторизован, **When** пользователь пытается удалить доску другого пользователя, **Then** возвращается ошибка с сообщением о том, что у пользователя нет прав на удаление этой доски

---

### Edge Cases

- Что происходит, когда пользователь пытается создать доску с названием, содержащим недопустимые символы (не буквы, цифры, пробелы или базовую пунктуацию)?
- Как система обрабатывает создание доски с очень длинным названием (более 255 символов)?
- Что происходит при попытке создать доску, когда у пользователя уже существует доска?
- Как система обрабатывает одновременное создание досок с одинаковым названием одним пользователем? (Решено: уникальное ограничение базы данных на user_id + name)
- Что происходит при удалении доски, в которой есть задачи, находящиеся в процессе выполнения?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система ДОЛЖНА позволять авторизованным пользователям создавать доски с уникальным названием в пределах их аккаунта
- **FR-002**: Система ДОЛЖНА автоматически создавать пять колонок при создании новой доски с названиями: "К выполнению", "В работу", "Возникла проблема", "Сделано", "Отмена" в указанном порядке
- **FR-003**: Система ДОЛЖНА валидировать название доски на наличие значения (не должно быть пустым)
- **FR-004**: Система ДОЛЖНА валидировать длину названия доски (максимум 255 символов)
- **FR-005**: Система ДОЛЖНА валидировать, что название доски содержит только разрешённые символы: буквы, цифры, пробелы и базовую пунктуацию (.,!?-:;)
- **FR-006**: Система ДОЛЖНА предотвращать создание досок с одинаковыми названиями для одного пользователя
- **FR-007**: Система ДОЛЖНА ограничивать количество досок на одного пользователя до 1
- **FR-008**: Система ДОЛЖНА возвращать ошибку при попытке создать доску, если у пользователя уже существует доска
- **FR-009**: Система ДОЛЖНА позволять авторизованным пользователям просматривать список всех своих досок
- **FR-010**: Система ДОЛЖНА отображать для каждой доски её название и количество задач в каждой колонке
- **FR-011**: Система ДОЛЖНА позволять авторизованным пользователям удалять созданные ими доски
- **FR-012**: Система ДОЛЖНА каскадно удалять все колонки и задачи при удалении доски
- **FR-013**: Система ДОЛЖНА предотвращать удаление досок, принадлежащих другим пользователям
- **FR-014**: Система ДОЛЖНА возвращать понятные сообщения об ошибках при неудачных операциях с досками

### Key Entities

- **Доска (Board)**: Представляет собой контейнер для организации задач. Содержит название, уникальный идентификатор, связь с владельцем (пользователем) и дату создания. Каждая доска имеет набор колонок. Название доски должно быть уникальным в пределах одного пользователя (unique constraint на user_id + name).
- **Колонка (Column)**: Представляет собой статус задач в рамках доски. Содержит название, уникальный идентификатор, связь с доской и порядок отображения (1-5). Колонки создаются автоматически при создании доски в порядке: "К выполнению" (1), "В работу" (2), "Возникла проблема" (3), "Сделано" (4), "Отмена" (5).
- **Пользователь (User)**: Владелец досок. Пользователь может создавать, просматривать и удалять только свои доски.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Пользователи могут создать новую доску с предопределёнными колонками менее чем за 30 секунд
- **SC-002**: Система поддерживает создание 1 доски на одного пользователя
- **SC-003**: 95% пользователей успешно создают свою первую доску с первой попытки без необходимости обращаться в поддержку
- **SC-004**: Список досок пользователя загружается менее чем за 1 секунду
- **SC-005**: Удаление доски с задачами выполняется менее чем за 2 секунды

## Assumptions

- Пользователь уже авторизован в системе (функциональность аутентификации реализована в рамках функции 001-user-auth)
- Задачи будут связаны с колонками и досками (функциональность управления задачами реализована в рамках функции 002-task-management)
- Названия колонок на русском языке соответствуют требованиям пользователей
- Пользователь может иметь только 1 доску
- Доски не могут быть переименованы или иметь настраиваемый набор колонок в рамках этой функции (это может быть добавлено в будущих функциях)

## Dependencies

- **001-user-auth**: Функция управления досками требует наличия авторизованного пользователя
- **002-task-management**: Функция управления досками интегрируется с функцией управления задачами для связи задач с колонками

## Clarifications

### Session 2026-03-02

- Q: Какой тип удаления досок должен быть реализован? → A: Жёсткое удаление (hard delete) - доска и все связанные данные удаляются из базы безвозвратно
- Q: Какой лимит количества досок на одного пользователя должен быть установлен? → A: 1 доска на пользователя
- Q: В каком порядке должны отображаться предопределённые колонки при создании доски? → A: К выполнению → В работу → Возникла проблема → Сделано → Отмена
- Q: Какие символы разрешены в названии доски? → A: Буквы, цифры, пробелы и базовая пунктуация (.,!?-:;)
- Q: Как система должна обрабатывать одновременное создание досок с одинаковым названием одним пользователем (race condition)? → A: Уникальное ограничение базы данных (unique constraint) на (user_id, name)

## Out of Scope

- Редактирование названия доски
- Создание пользовательских колонок с произвольными названиями
- Изменение порядка колонок
- Переименование колонок
- Удаление колонок
- Совместное использование досок между пользователями
- Копирование досок
- Архивирование досок
- Шаблоны досок
- Экспорт/импорт досок
- Восстановление удалённых досок (hard delete безвозвратен)

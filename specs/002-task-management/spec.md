## User Scenarios & Testing *(mandatory)*

### User Story 1 - Создание новой задачи (Priority: P1)

Пользователь может создать новую задачу, указав название, описание, срок исполнения и начальный статус. Задача сохраняется в системе и становится доступной для дальнейшей работы.

**Why this priority**: Это базовая функциональность без которой система управления задачами не может работать. Создание задач - это первичный сценарий использования.

**Independent Test**: Можно полностью протестировать путём создания одной задачи через интерфейс и проверки её сохранения в системе. Доставляет ценность - пользователь может зафиксировать задачу в системе.

**Acceptance Scenarios**:

1. **Given** пользователь авторизован в системе, **When** пользователь открывает форму создания задачи и заполняет все обязательные поля (название, описание, срок исполнения, статус), **Then** задача успешно создаётся и сохраняется в системе
2. **Given** пользователь авторизован в системе, **When** пользователь создаёт задачу с описанием длиной ровно 5000 символов, **Then** задача успешно создаётся и всё описание сохраняется
3. **Given** пользователь авторизован в системе, **When** пользователь создаёт задачу с описанием длиной 5001 символ, **Then** система показывает ошибку валидации и не сохраняет задачу

---

### User Story 2 - Просмотр списка задач (Priority: P2)

Пользователь может просмотреть список всех созданных задач с их основными параметрами (название, статус, срок исполнения).

**Why this priority**: После создания задач пользователю необходимо видеть их список для управления и отслеживания прогресса.

**Independent Test**: Можно протестировать путём создания нескольких задач и проверки их отображения в списке. Доставляет ценность - пользователь видит все свои задачи в одном месте.

**Acceptance Scenarios**:

1. **Given** в системе существуют несколько задач с разными статусами, **When** пользователь открывает список задач, **Then** отображаются все задачи с их текущими статусами и сроками исполнения
2. **Given** в системе нет задач, **When** пользователь открывает список задач, **Then** отображается пустой список с сообщением о том, что задачи отсутствуют

---

### User Story 3 - Изменение статуса задачи (Priority: P3)

Пользователь может изменить статус существующей задачи на один из предопределённых статусов.

**Why this priority**: Изменение статуса необходимо для отслеживания прогресса выполнения задач, но не является критичным для базового функционала.

**Independent Test**: Можно протестировать путём создания задачи и последовательного изменения её статуса. Доставляет ценность - пользователь может отражать текущее состояние задачи.

**Acceptance Scenarios**:

1. **Given** существует задача со статусом "к выполнению", **When** пользователь изменяет статус на "в работу", **Then** статус задачи обновляется и сохраняется
2. **Given** существует задача со статусом "сделано", **When** пользователь пытается изменить статус на "к выполнению", **Then** статус задачи обновляется (разрешено изменение в любом направлении)

---

### Edge Cases

- Что происходит, когда пользователь пытается создать задачу без названия?
- Как система обрабатывает создание задачи с пустым описанием?
- Система разрешает создание задач с датой исполнения в прошлом без предупреждений
- Как система обрабатывает создание задачи с некорректным форматом даты?
- Что происходит при попытке установить статус, не входящий в предопределённый список?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система ДОЛЖНА позволять авторизованным пользователям создавать задачи
- **FR-002**: Система ДОЛЖНА требовать обязательного заполнения названия задачи при создании
- **FR-003**: Система ДОЛЖНА позволять указывать описание задачи длиной от 0 до 5000 символов
- **FR-004**: Система ДОЛЖНА отклонять описание задачи длиной более 5000 символов с сообщением об ошибке
- **FR-005**: Система ДОЛЖНА позволять указывать срок исполнения задачи в формате даты (YYYY-MM-DD)
- **FR-006**: Система ДОЛЖНА позволять выбирать статус из предопределённого списка: "к выполнению", "в работу", "возникла проблема", "сделано", "отмена"
- **FR-007**: Система ДОЛЖНА запрещать указание статуса, не входящего в предопределённый список
- **FR-008**: Система ДОЛЖНА сохранять созданную задачу в хранилище данных
- **FR-009**: Система ДОЛЖНА позволять просматривать список всех созданных задач пользователя без сортировки (в порядке хранения в БД)
- **FR-010**: Система ДОЛЖНА отображать для каждой задачи: название, статус, срок исполнения
- **FR-011**: Система ДОЛЖНА позволять изменять статус существующей задачи
- **FR-012**: Система ДОЛЖНА ограничивать доступ пользователя только к его собственным задачам
- **FR-013**: Система ДОЛЖНА запрещать пользователю просматривать или изменять задачи других пользователей

### Key Entities

- **Задача (Task)**: Представляет единицу работы, которую необходимо выполнить. Содержит название, описание, срок исполнения и текущий статус. Принадлежит пользователю, который её создал. Только владелец задачи может просматривать и изменять её.
- **Статус задачи (Task Status)**: Предопределённое значение, отражающее текущее состояние задачи. Возможные значения: "к выполнению", "в работу", "возникла проблема", "сделано", "отмена".

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Пользователь может создать задачу с корректными данными менее чем за 30 секунд
- **SC-002**: Система поддерживает создание и хранение не менее 10 000 задач без снижения производительности
- **SC-003**: 95% пользователей успешно создают задачу с первого раза без ошибок валидации
- **SC-004**: Список задач загружается и отображается менее чем за 2 секунды при наличии до 100 задач
- **SC-005**: Изменение статуса задачи выполняется и сохраняется менее чем за 1 секунду

## Assumptions

- Пользователи уже авторизованы в системе (функция авторизации реализована отдельно)
- Срок исполнения задачи указывается в формате даты (YYYY-MM-DD) без времени
- Пустое описание задачи допустимо (описание не является обязательным полем)
- Статус задачи может быть изменён в любом направлении (нет ограничений на переходы между статусами)
- Задачи привязаны к пользователям, которые их создали
